# Networking

GuildServer uses Docker networking to connect deployed application containers, manage traffic routing through Traefik, and provide external access through custom domains. This page explains the networking architecture and how containers communicate.

## Network Architecture

```
                    Internet
                       |
                 +-----------+
                 |  Traefik  |  (ports 80, 443, 8080)
                 +-----------+
                       |
          +------------+------------+
          |                         |
   guildserver-network       guildserver (external)
   (default compose net)     (deployed app containers)
          |                         |
   +------+------+          +------+------+
   | postgres     |          | gs-myapp-*  |
   | redis        |          | gs-blog-*   |
   | api          |          | gs-api-*    |
   | web          |          | my-db       |
   +--------------+          +-------------+
```

Traefik sits on **both** networks, bridging infrastructure services and deployed applications.

## The `guildserver` Docker Network

The `guildserver` network is an **external bridge network** shared between the platform infrastructure and all deployed application containers. It is the backbone of GuildServer's networking model.

### Creating the Network

The network must be created manually before starting the platform:

```bash
docker network create guildserver
```

This is declared as an external network in `docker-compose.yml`:

```yaml
networks:
  guildserver:
    external: true
    name: guildserver
```

### Why an External Network?

The network is external (not managed by Docker Compose) because:

1. **Deployed containers are not part of the Compose stack** -- They are created dynamically by the API when users deploy applications via the `deployContainer()` function.
2. **Traefik needs shared network access** -- Traefik must be on the same network as the containers it routes to.
3. **Lifecycle independence** -- The network persists even if `docker compose down` is run, so deployed apps remain reachable.

### Network Initialization at Deploy Time

When the API deploys a container, it first ensures the network exists by calling `ensureNetwork()`:

```typescript
const NETWORK_NAME = "guildserver";

export async function ensureNetwork(): Promise<void> {
  try {
    const network = docker.getNetwork(NETWORK_NAME);
    await network.inspect();
  } catch {
    logger.info(`Creating Docker network: ${NETWORK_NAME}`);
    await docker.createNetwork({
      Name: NETWORK_NAME,
      Driver: "bridge",
    });
  }
}
```

If the network does not exist, it is created as a bridge network. This is a safety mechanism -- normally the network is created once during initial setup.

## The `guildserver-network` (Default Compose Network)

The default Compose network (`guildserver-network`) connects the core platform services:

```yaml
networks:
  default:
    name: guildserver-network
```

Services on this network:
- **postgres** (`guildserver-postgres`) -- Primary database
- **redis** (`guildserver-redis`) -- Cache and queue broker
- **api** (`guildserver-api`) -- Backend tRPC server
- **web** (`guildserver-web`) -- Next.js frontend
- **traefik** (`guildserver-traefik`) -- Also connected to the `guildserver` network

## How Containers Communicate

### Service-to-Service (Within Compose)

Core services communicate using Docker's built-in DNS. Each service is reachable by its service name within the `guildserver-network`:

```bash
# API connects to PostgreSQL using the service name "postgres"
DATABASE_URL=postgresql://guildserver:password123@postgres:5432/guildserver

# API connects to Redis using the service name "redis"
REDIS_URL=redis://redis:6379
```

Docker resolves `postgres` to the container's IP address on the `guildserver-network` automatically.

### Deployed App Communication

Deployed application containers are on the `guildserver` network and can communicate with each other by container name. The naming convention is:

```
gs-{appName}-{first8CharsOfDeploymentId}
```

Generated by:

```typescript
const CONTAINER_PREFIX = "gs";

function containerName(appName: string, deploymentId: string): string {
  const shortId = deploymentId.slice(0, 8);
  return `${CONTAINER_PREFIX}-${appName}-${shortId}`;
}
```

For example: `gs-my-api-a1b2c3d4`

:::info
Container names change with each deployment because the deployment ID changes. If your application needs to reference another application by hostname, use the domain name routed through Traefik instead of the container name for stable addressing.
:::

## Port Mapping

### Dynamic Host Port Assignment

Every deployed container receives a **dynamic host port** in the ephemeral range (10000--60000). The deployment service scans all running containers to find an available port:

```typescript
async function findAvailablePort(): Promise<number> {
  const MIN_PORT = 10000;
  const MAX_PORT = 60000;
  const usedPorts = new Set<number>();

  // Get ports used by existing containers
  const containers = await docker.listContainers({ all: true });
  for (const c of containers) {
    if (c.Ports) {
      for (const p of c.Ports) {
        if (p.PublicPort) usedPorts.add(p.PublicPort);
      }
    }
  }

  // Find a random available port (up to 100 attempts)
  for (let attempt = 0; attempt < 100; attempt++) {
    const port = MIN_PORT + Math.floor(Math.random() * (MAX_PORT - MIN_PORT));
    if (!usedPorts.has(port)) return port;
  }

  throw new Error("No available ports found");
}
```

The mapping is configured in the container's `HostConfig`:

```typescript
HostConfig: {
  PortBindings: {
    [`${servicePort}/tcp`]: [{ HostPort: String(hostPort) }],
  },
  NetworkMode: NETWORK_NAME,
}
```

### Port Assignment Flow

```
1. App specifies containerPort (e.g., 3000)
   OR auto-detected from image (e.g., nginx -> 80)

2. GuildServer finds available hostPort (e.g., 34521)

3. Docker maps:  34521 (host) -> 3000 (container)

4. Direct access:  http://localhost:34521
   Domain access:  http://my-app.localhost -> Traefik -> container:3000
```

### Infrastructure Port Mappings

The core platform services use fixed port mappings to avoid conflicts with common local services:

| Service    | Container Port | Host Port | Notes |
|------------|---------------|-----------|-------|
| Traefik    | 80, 443, 8080 | 80, 443, 8080 | Standard HTTP/HTTPS + dashboard |
| PostgreSQL | 5432          | 5433      | Offset to avoid local PG conflicts |
| Redis      | 6379          | 6380      | Offset to avoid local Redis conflicts |
| API        | 4000          | 4000      | Direct mapping |
| Web        | 3000          | 3000      | Direct mapping |

## Traefik Routing

Traefik is the primary way external users access deployed applications. The routing flow:

```
1. User request: http://my-app.example.com
2. DNS resolves to the GuildServer server IP
3. Traefik receives on port 80 (or 443 for HTTPS)
4. Matches Host(`my-app.example.com`) rule from container labels
5. Forwards to container on the guildserver network
6. Hits container's internal port (e.g., 3000)
```

:::tip
Traefik uses the container's **internal** service port (not the host port) because it communicates directly through the Docker bridge network. The host port is only used for direct browser access during debugging.
:::

### Label-Based Routing

Each deployed container gets Traefik labels that define its routing:

```
traefik.enable=true
traefik.http.routers.my-app.rule=Host(`my-app.localhost`)
traefik.http.services.my-app.loadbalancer.server.port=3000
```

See the [Traefik documentation](./traefik.md) for complete details on label configuration, TLS, and multi-domain setups.

## External Access Patterns

### Custom Domains (Production)

1. User adds a custom domain (e.g., `app.example.com`) to their application in the dashboard.
2. User configures DNS to point `app.example.com` to the GuildServer server's IP address.
3. GuildServer attaches Traefik labels with the domain's `Host` rule to the container.
4. Traefik provisions an SSL certificate via Let's Encrypt (HTTP-01 challenge).
5. Traffic flows: `app.example.com -> Traefik -> container`.

### Localhost Domains (Development)

During development, applications get `.localhost` domains (e.g., `my-app.localhost`). Modern browsers resolve `*.localhost` to `127.0.0.1` automatically without `/etc/hosts` changes. These domains use plain HTTP (no TLS).

### Direct Port Access (Debugging)

The dynamically assigned host port provides direct access without Traefik:

```
http://localhost:{hostPort}
```

The host port is returned in the deployment response and visible in the application dashboard.

### Database External Access

Managed databases can optionally expose an `externalPort` for external client connections (defined on the `databases` table):

```sql
external_port INTEGER
```

When set, tools like pgAdmin, DBeaver, or TablePlus can connect directly:

```
Host: localhost
Port: {externalPort}
Database: {databaseName}
User: {username}
Password: {password}
```

:::warning
Exposing database ports externally should only be done in development or behind a firewall. In production, prefer connecting through the Docker network using the container hostname.
:::

## Network Security

### Container Restart Policy

All deployed containers use the `unless-stopped` restart policy:

```typescript
RestartPolicy: { Name: "unless-stopped", MaximumRetryCount: 0 }
```

This ensures containers automatically restart after crashes or server reboots, but stay stopped if manually stopped by the user.

### Docker Socket Access

Traefik has **read-only** access to the Docker socket:

```yaml
volumes:
  - /var/run/docker.sock:/var/run/docker.sock:ro
```

The `:ro` flag prevents Traefik from modifying containers -- it can only observe them for routing purposes.

### Container Isolation

Containers on the `guildserver` bridge network can communicate freely with each other by default. This is by design -- it allows applications to connect to their managed databases and other services within the platform. For stronger isolation between tenants, consider deploying to separate [Kubernetes clusters](./kubernetes.md) with namespace-based isolation.

## Troubleshooting

### Container unreachable from Traefik

1. Verify the container is on the `guildserver` network: `docker network inspect guildserver`.
2. Check that `traefik.enable=true` is present in the container labels: `docker inspect <container>`.
3. Confirm the Host rule matches the requested domain exactly.
4. Review Traefik logs: `docker compose logs -f traefik`.

### Port conflict at deploy time

If `findAvailablePort()` fails, too many containers may be running. Check container count:

```bash
docker ps -q | wc -l
```

The ephemeral range (10000--60000) supports approximately 50,000 containers. Clean up old deployments to free ports.

### DNS resolution failures between containers

Docker DNS only works within the same network. Verify both containers are on the `guildserver` network:

```bash
docker inspect <container> --format '{{json .NetworkSettings.Networks}}'
```
